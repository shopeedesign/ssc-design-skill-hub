# Skill 撰写模板（教程版）

> **用途**：作为团队内撰写 Codex / Cursor / Claude Code 等工具可用的 **Skill**（通常以 `SKILL.md` 交付）时的结构清单与填空模板。  
> **原则**：先定义「谁在什么情况下得到什么」，再写「怎么做」；触发条件、边界、输入输出写清楚，比堆长规则更重要。

---

## 一、文档头部（YAML Frontmatter）

在 `SKILL.md` 最上方使用 YAML 元数据（具体语法以目标平台为准；以下为常见写法）：

```yaml
---
name: your-skill-id
description: >-
  用 2～6 行说明：本 skill 解决什么问题、面向谁、默认产出是什么。
  必须包含「自然语言可检索」的触发词（用户会怎么开口找你）。
  写清不适用边界（可选一行），减少误触发。
---
```

**填写要点**


| 字段            | 要求                                       |
| ------------- | ---------------------------------------- |
| `name`        | 小写、短横线、与目录名一致，便于机器引用。                    |
| `description` | 既是「说明书摘要」，也是**语义触发**素材：用户口头说法应能命中这里的关键词。 |


---

## 二、Skill 一句话定位

用 **1～2 句**回答：这是谁、在什么场景下的、什么类型的助手。

**模板句式**

> 本 Skill 是面向 **[角色]** 的 **[能力类型]** 助手，当用户 **[触发情境]** 时，帮助其 **[核心产出]**。

**自检**

- 读完这两句，新人能否判断「我该不该用这个 skill」？

---

## 三、Skill 定位与边界（强烈建议单独成节）

### 3.1 适用场景（列举 3～6 条）

- …
- …

### 3.2 不适用场景（列举 2～5 条）

明确写出「用户更该去用别的 skill / 别的工具」的情况，例如：

- …
- …

**自检**

- 是否与团队内其它 skill（如 PRD 理解、设计自查、Figma 数据审查等）**分工清晰、不重叠打架**？

---

## 四、触发条件（机器 + 人都要懂）

列出用户**说什么 / 贴什么**时应进入本 Skill，例如：

- 用户明确提到：`…`、`…`
- 用户提供了：`链接` / `文件` / `截图` / `结构化 ID` 等
- 用户表达的意图是：`…`

**可选：反向触发**

- 仅当用户说「随便聊聊」且未提供 `…` 时，**不要**进入本流程。

---

## 五、角色定义（Persona）

用简短条目定义模型**口吻、立场、优先级**，例如：

- 你是 **…领域** 的 **…角色**，优先关注 **…**
- 先 **理解 / 收集** 再 **判断 / 输出**，避免跳步
- 对用户数据与权限保持 **保守、可解释**

---

## 六、输入与前置条件

### 6.1 用户需要提供什么


| 输入类型        | 是否必填    | 说明  |
| ----------- | ------- | --- |
| 例如：Figma 链接 | 必填 / 可选 | …   |
| 例如：业务背景说明   | 可选      | …   |


### 6.2 环境与依赖（若有）

- 是否需要 **MCP / 插件 / 本地凭据**（路径、文件名、环境变量名写清楚）
- 若依赖缺失，**应停止并提示补全**，而不是猜测或绕过

### 6.3 范围约定（若有）

- 多页面 / 多文件时，默认只处理 **用户指定范围**；全量处理前须 **显式确认**

---

## 七、执行流程（分阶段、可核对）

将工作流拆成 **Phase 1 → Phase 2 → …**，每阶段写清：

1. **目的**（这一阶段解决什么）
2. **动作**（读取、提问、调用工具、归纳）
3. **退出条件**（什么算完成、什么要停下来问用户）

**模板结构示例**

```markdown
### Phase 1：…
1. …
2. …

### Phase 2：…
1. …
```

**自检**

- 是否存在「未收集关键信息就输出终稿」的步骤？若有，补上 **门禁**（必须先问 / 必须先读）。

---

## 八、输出规范（强烈建议给「可直接粘贴」的模板）

给出**默认输出结构**（Markdown 标题层级、表格字段、分级标签等），例如：

- 一级标题、二级标题固定顺序
- 列表项每条包含：**现象 → 影响 → 建议**（按需）
- 严重度标签统一：`严重` / `一般` / `优化`（或你们团队统一枚举）

**可选**

- 提供 **「推荐用户复制使用的提问例」** 2～3 条

---

## 九、质量与安全守则（Guardrails）

根据 skill 类型勾选撰写（不必每条都有）：

- **权限**：不得在未授权时访问内网系统；缺凭据时明确提示路径与字段名
- **写操作**：批量修改、发版、删数据等，必须 **用户确认** 后再执行
- **诚实性**：无法访问文件 / 无法调用工具时，明确说明限制，**禁止编造**页面内容
- **隐私**：不鼓励粘贴真实手机号、真实密钥；若已出现，提醒脱敏
- **一致性**：术语、状态枚举、单位与团队设计规范对齐

---

## 十、与其它资产的关系（可选）

说明本仓库内还应包含什么（若适用）：


| 文件 / 目录       | 作用                            |
| ------------- | ----------------------------- |
| `README.md`   | 给人看的快速上手、示例指令、FAQ             |
| `reference/…` | 长清单、细则、分册知识库，由模型在 deep 阶段按需读取 |
| `SKILL.md`    | 给模型看的编排与门禁（宜控制篇幅）             |


---

## 十一、自测清单（发布前勾一遍）

- `description` 里能否 **不靠内部黑话** 被用户搜到？
- **触发**与**不触发**是否都写了？
- **输入缺省**时，有没有默认追问或默认安全行为？
- **输出模板**是否可直接复制到 Confluence / PRD / 评审记录？
- 是否声明 **工具 / MCP / 凭据** 依赖及失败时的提示语？
- 与兄弟 skill 的 **边界** 是否有一句交叉引用？

# 附录：Skill撰写说明

下面按「第一次写 skill」的最短路径讲解。假设目标是把 skill 安装到本机目录（例如 Codex：`~/.codex/skills/<skill-id>/SKILL.md`，Cursor 同理对应 skills 目录），安装后用户在对话里用自然语言即可触发。

## A. 先想清 4 个问题（比先写正文重要）

1. **谁会用？**（设计师 / 产品 / 研发 / 运营…）
2. **在什么时刻用？**（开工前、评审前、交付前、排障时…）
3. **用户说完哪句话，你就该接手？**（把这些话写进 YAML `description`）
4. **交付物长什么样？**（给用户一个「复制即用」的输出模板）

这 4 个问题对应模板里的：**定位、触发、输出**。写顺了，skill 就成功了一半。

## B. 最小可用文件（MVP）

第一次写建议 **只维护一个 `SKILL.md`**，不要一上来拆很多 reference 文件。等到规则超过约 **300～500 行**、或明显有「索引 + 分册」需求时，再拆 `reference/`。

**MVP 推荐目录**

```text
your-skill-id/
  SKILL.md          # 必填：编排 + 触发 + 输出模板
  README.md       # 强烈建议：给人类同事的「怎么用」
```

## C. 推荐撰写顺序（照着填，不容易漏）

1. 写 YAML：`name` + `description`（触发词写全）
2. 写 **不适用场景**（帮你减少 50% 误用投诉）
3. 写 **输入与前置条件**（尤其是凭据、链接、权限）
4. 写 **分阶段流程**（每阶段 3～7 步为宜，超过则考虑拆 reference）
5. 写 **输出模板**（用 Markdown 框死结构）
6. 写 **安全守则**（有没有写操作？要不要确认？）
7. 写 README：用 3 个「用户原话示例」演示怎么调用

## D. 怎样算「完全可用」？

满足下面 **5 条** 就可以给同事试用了：

1. **能触发**：同事用日常说法说需求，模型能稳定进入本 skill 的流程。
2. **不串台**：和团队其它 skill 的边界清楚，误触发时有 **一句话引导** 到正确 skill。
3. **不缺门禁**：缺关键输入、缺凭据、缺工具时，会 **停下来说明缺什么**，而不是胡编。
4. **产出可执行**：输出结构固定，读者看完知道 **下一步改什么 / 问谁 / 补什么材料**。
5. **可安装可发现**：`SKILL.md` 路径正确；README 里有一条 **「装好后怎么试」** 的示例指令。

## E. 常见翻车点


| 现象         | 原因               | 改法                       |
| ---------- | ---------------- | ------------------------ |
| 经常误触发      | `description` 太泛 | 收紧触发词 + 写清不适用            |
| 输出每次格式不同   | 没给固定模板           | 给 Markdown 骨架，要求逐段填      |
| 一上来就给结论    | 缺「收集背景」阶段        | 加 Phase 0 / Phase 1 强制提问 |
| 引用一大段内部政策  | SKILL 过长         | 搬到 `reference/`，正文只保留索引  |
| 写了工具调用但没环境 | 依赖没声明            | 在「前置条件」写明路径与失败提示         |


## F. 从「能跑」到「好用」的迭代建议

1. 找 3 个真实用户各试 **3 次**，收集「误触发 / 漏问 / 输出不可用」三类问题。
2. 每次迭代只改 **一类问题**（触发、流程、输出三选一），避免一次大改无法归因。
3. 把「用户原话」**反哺**进 `description` 与 README 示例，skill 会越来越「像团队里的人」。
