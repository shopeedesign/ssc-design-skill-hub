第 01 章 · AI 工作机制与上下文工程

Skill 的本质

按需注入的可复用的AI工作流程文档
Skill 的完整定义

Skill 是存储于 .md 文件的工作流程文档，包含角色定义、执行流程与输出规范三大要素。任务触发时按需注入到 AI 的上下文窗口，为其提供系统性的专业执行指导。

01
持久化存储
内容一次编写，永久有效。无需在每次会话中重新声明任务要求。
02
按需注入
Skill 仅在相关任务触发时加载至上下文窗口，不常驻内存，制造最佳 Token 效率。
03
可复用与可共享
单个 Skill 可被个人、团队、跨项目共享复用，也支持版本控制与独立迭代。
「Skill」 与「临时提示词」的核心差异
维度临时提示词Skill
存在方式仅当次对话内有效持久化存储，永久可用
一致性高度依赖操作者描述质量固定流程与输出规范，结果可预期
可维护性随时修改困难，内容难管理文件化管理，已版本控制、独立迭代
团队协作个人经验难共享文件化共享，团队可统一使用
Token 效率每次对话重复录入描述仅在需要时按需注入
更新方式执行时手动内嵌描述独立文件单独维护，不影响其他流程

快速决策：我应该用哪种工具？

根据任务特征选择合适的工具，三者配合使用效果最佳
任务场景描述推荐工具关键原因
全局语气风格、回复语言、代码风格约定，希望每次对话均自动生效
Rules
全局常驻，每次对话均需生效，无需主动触发
告知 AI 项目架构背景、团队规范、角色定义
Rules
每次对话均需完整背景，无需重复声明
封装一个多步骤的复杂操作流程，如设计稿自查、提案撰写
Skills
需要顺序执行多阶段流程，只在特定任务时才需要
同一任务类型需多次重复执行，希望每次执行质量一致
Skills
持久化工作流程文档，结果可预期，不依赖个人记忆
危险命令执行前必须强制人工确认，不允许绕过
Hooks
系统层强制拦截，不依赖 AI 理解，不可绕过
文件保存后自动运行格式化、提交前自动扫描敏感信息
Hooks
事件驱动完全自动化，无需主动调用，不经过 AI
希望同时具备全局语气约定 + 共享操作流程 + 安全拦截
Rules + Skills + Hooks
三层各司其职，配合使用实现最佳效果

Skill+Rules架构优势

为什么要进行架构拆分
01Token 节约
旧每次对话全量加载（可能超 100KB）
新仅加载 1-2 个当前任务所需 Skill
02消除指令干扰
旧多个流程共存导致指令互相干扰

03无限扩展
旧系统提示词容量有上限，越大越混乱

04模块化维护

06团队协作友好
旧系统提示词属于个人设定，难以按需调整
新文件化共享，团队共用相同工作流
07触发更精准
旧所有流程始终展示，无关任务多次冗余

第 04 章 · 案例解析：design-selfcheck

design-selfcheck 的 4 阶段执行流程

任务触发后，AI 将严格按照此流程进行，每个阶段有明确的输入要求与输出预期
→
→
→
Phase 1
感知设计稿
  • 请就设计稿分享截图或 Figma 链接
  • 若无法直接访问 Figma，则基于截图推断（标注推断来源）
  • 内置引导话术：「可以分享一下...」
Phase 2
收集需求背景
  • 及时收集 PRD / 业务需求文档
  • [强制约束] 在没有足够背景信息时，AI 不得开始分析
  • 内置引导对话：方式 A（对话收集）或方式 B（PRD 上传）
Phase 3
执行自查分析
  • 按需读取 reference/checklist/ 目录下的清单模块
  • 对照设计稿逐一根据模块进行深度分析
  • 按缺陷严重程度对问题进行分级
Phase 4
输出自查报告
  • 固定五段输出结构：需求摘要、设计推导、缺陷分析、其他方向、综合建议
  • 以分析报告方式输出，提升可对照性
  • 禁止平铺呈现层级 / 全文平铺

第 04 章 · 案例解析：design-selfcheck

写作要点：元数据、角色与约束

每个要点均对应“AI 在哪里会犯错”的一种假设，并给出明确的应对策略
要点 01
Description 三合一
领域定位 + WHAT + WHEN + 触发词
痛点
如果 Description 模糊，AI 不知该在何时加载此 Skill
写作要点
B 端电商供应链产品（领域）、设计稿系统性自查（WHAT）、Figma 链接或「设计自查」时触发（WHEN + 触发词）
▶ YAML >- 格式支持多行描述，允许 1000+ 字的详尽说明
要点 02
赋予 AI 专业角色
将 AI 定义为领域专家
痛点
AI 以通用对话视角进入判断，而非专家审查视角检出问题
写作要点
「你是一位资深 B 端产品设计审查专家，专注于电商供应链领域，局限在交付前设计稿的系统性审查」
▶ 公式：身份 + 领域 + 责任范围；层层缩小 AI 发挥空间
要点 03
强制约束语言
双重强调，标题 + 正文
痛点
AI 在步骤 Phase 2 缺少需求背景时仍试图开始分析
写作要点
标题标注「不得跳过此步骤」，正文再强调「必须先收集需求背景再执行」。「不得」强于「尽量」，等于确确实实设置程序内的确认标志位
▶ 最高级别：标题强调 + 正文强调 + 术语选择

第 04 章 · 案例解析：design-selfcheck

写作要点：流程控制、知识库与输出封控

AI 在某个环节上自由发挥的空间越小，Skill 越好用。下列四个要点展示如何控制 AI 的执行边界
要点 04
话术模板内嵌
高频语句放入代码块
痛点
AI 话术不一致，引导设计师提供素材的方式容易失控
写作要点
Phase 1/2 各有一段可直接复制的引导话术：「分享设计稿截图或 Figma 链接，或导出 JSON」。模板确保 AI 原文复现而非自行改写。
YAML 代码块内可拾取的话术模板→确保 AI 正确引用
要点 05
条件分支 + 兜底路径
认可输入不符预期的各种情况
痛点
输入不符预期时 AI 会卡住或强行执行
写作要点
Phase 2 设计 A/B 两条路径：对话收集 vs PRD 上传。Phase 1 无法访问 Figma 时有「基于描述推断（标注来源）」兜底路径。
每个 Phase 都考虑「实现不了怎么办」并在 Skill 中写明
要点 06
渐进式知识库
主文件入口 + 子文件细节分离
痛点
将所有清单内嵌 SKILL.md 会导致 Token 爆用
写作要点
主文件仅指向 reference/checklist/README.md 索引；6 个清单模块存放在独立文件中，按需读取。上下文有限时指定最小子集。
主文件 > 500 行就拆分；链接层级不超过一层
要点 07
输出模板 + 禁令封控
正向模板 + 负向禁令双封锁
痛点
输出结构漂移、内部编号暴露给用户
写作要点
固定五段输出：需求背景摘要、设计推导、缺陷分析、其他方向、综合建议。禁令：不得展示清单内部编号、不得平铺平级。
模板正向封锁 + 禁令负向封锁 = AI 输出最可控

第 05 章 · 编写规范与实操技巧

SKILL.md 结构解剖：三大组成部分

一个完整的 SKILL.md 文件由 Frontmatter、Body 和附加资料三部分构成。Frontmatter 是 AI 判断是否加载的唯一依据
最关键
①Frontmatter（必填）

提供元数据，是 AI 判断是否加载此 Skill 的唯一依据。

  • name：小写 + 连字符，最多 64 字符
  • description：领域 + WHAT + WHEN + 触发词，第三人称，最多 1024 字
  • 不可省略，排序与字段名必须准确
主体
②Body（必需）

定义 AI 的角色、触发条件与执行流程。

  • 包含角色定义、触发条件、执行流程、注意事项
  • 建议主文件保持 < 500 行
  • 每个阶段目标明确，无冗余内容
渐进式
③附加资料（可选）

按需加载的模板、清单与示例，避免主文件臃肿。

  • 大型清单、模板、示例拆到子文件
  • reference/ 目录存放，主文件提供入口链接
  • 链接层级不超过一层，避免过深嵌套
完整目录示例

第 05 章 · 编写规范与实操技巧

Description 写法三要则：最关键的字段

Description 注入系统提示词，不是对用户说话。写得越具体， AI 加载时机越准确
原则 01
第三人称叙述
不是对用户说话，是对系统说明
✖ 不建议写法
这个 Skill 帮你审查设计稿存在的问题
✔ 建议写法
This skill performs systematic design review for B2B e-commerce product designers before delivery. → 属于系统，第三人称
原则 02
WHAT + WHEN 全包含
同时描述做什么、何时触发
✖ 不建议写法
设计自查工具，用于审查设计稿
✔ 建议写法
对B端产品师的数字系统设计自查工作流（WHAT），在分享 Figma 链接或提及设计自查时触发（WHEN）
原则 03
包含具体触发词
触发词越具体，加载越准确
✖ 不建议写法
用户说要审查设计时触发
✔ 建议写法
分享 Figma 链接 / 「帮我查一下设计稿」 / 「进入评审模式」 / 「设计自查」 等语言时触发

第 05 章 · 编写规范与实操技巧

4 种常用写作模式

根据任务类型选择合适的模式；可组合使用，design-selfcheck 同时应用了这4种模式
模板模式Template Pattern

提供固定结构的输出格式，确保 AI 每次输出一致

应用案例: design-selfcheck Phase 4：五段固定输出模板（需求摘要、设计推导、缺陷分析、其他方向、综合建议）
适用：高频执行且输出需要一致性的任务
流程模式Workflow Pattern

将任务拆分为有序执行的多个阶段，每阶段有明确输入与输出

应用案例: design-selfcheck Phase 1→4：感知设计稿→收集需求→执行自查→输出报告
适用：多步骤复杂任务，顺序不可颠倒
条件模式Conditional Pattern

根据输入条件选择不同路径，包括兜底场景的处理

应用案例: design-selfcheck Phase 2：方式 A（对话收集）vs 方式 B（PRD 直接上传）
适用：输入来源不确定、或需处理多种异常情况
强制约束模式Constraint Pattern

明确列出 AI 不得做的事项，封锁确切的分析边界

应用案例: design-selfcheck 注意事项第 2 条：「严禁在报告中以任何形式出现条目编号（如 "2.3 信息架构"、"P1 必查项" 等），所有结论必须用自然语言直接表达」
适用：容易发生漂移或过度发挥的任务
